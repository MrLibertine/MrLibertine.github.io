Unity 架构与设计速查

2019-06-07 1.1k 字
Unity C#

单例的几种写法

MonoBehaviour 单例

public class GameManager : MonoBehaviour
{
    public static GameManager Instance { get; private set; }

    void Awake()
    {
        if (Instance != null && Instance != this)
        {
            Destroy(gameObject);
            return;
        }
        Instance = this;
        DontDestroyOnLoad(gameObject);
    }
}

注意：多线程环境需加锁，但 Unity API 只能在主线程调用，通常不需要线程安全。

纯 C# 单例（无 MonoBehaviour 依赖）

public class ConfigManager
{
    public static ConfigManager Instance { get; } = new ConfigManager();
    private ConfigManager() { }
}

适用场景：纯数据管理、不需要 Update 周期的类。

事件系统 / 消息中心

public class EventBus
{
    public static EventBus Instance = new EventBus();

    private Dictionary<string, Action<object>> events = new();

    public void Subscribe(string eventId, Action<object> callback)
    {
        if (events.ContainsKey(eventId))
            events[eventId] += callback;
        else
            events[eventId] = callback;
    }

    public void Unsubscribe(string eventId, Action<object> callback)
    {
        if (events.ContainsKey(eventId))
            events[eventId] -= callback;
    }

    public void Publish(string eventId, object data = null)
    {
        if (events.TryGetValue(eventId, out var action))
            action?.Invoke(data);
    }
}

关键注意：场景切换前确保取消订阅，否则接收者被销毁后回调残留 → 空引用异常或内存泄漏。在 OnDisable / OnDestroy 中 Unsubscribe。

UnityEvent 对比：UnityEvent 可在 Inspector 面板拖拽绑定，适合编辑器配置，但序列化后的引用在跨场景时需要留意。

对象池设计要点

（配套内存管理篇的简易实现，此处补充工程化考量）

  • 池容量上限：设置最大值防无限膨胀；超过上限时 Return 直接 Destroy。
  • 定时收缩：每 N 秒检查一次，超过闲置时间的对象释放掉。
  • 分级池：小型对象用通用池，大型或初始化昂贵的对象用专用池（如复杂 UI 面板）。
  • 自动回收：在池对象上挂 ReturnToPool 组件，OnDisable 时自动归还。
public class PoolObject : MonoBehaviour
{
    public System.Action<PoolObject> OnReturn;
    void OnDisable() => OnReturn?.Invoke(this);
}

UI 架构：MVP 与 MVC 落地

MVC（Model-View-Controller）

层 职责 Unity 中的典型实现
Model 数据与逻辑 纯 C# 类，ScriptableObject
View 显示 UI 预制体 + 绑定脚本
Controller 协调输入 → Model → View MonoBehaviour 的中间脚本

优点：逻辑与显示分离。缺点：View 和 Controller 容易耦合。

MVP（Model-View-Presenter）

  • View 只负责”画”，不持有任何逻辑。
  • Presenter 持有 View 接口引用，操作 View 的属性（Text、Image 等）。
  • View 通过接口 IView 暴露可被 Presenter 操控的元素。
  • 优点：View 与逻辑彻底解耦，可测试。

实际选择：小项目 MVP 过度设计，用精简的 Controller 处理即可。复杂 UI 面板（商城、背包）MVP 能显著降低维护成本。

ScriptableObject 做配置表

[CreateAssetMenu(fileName = "EnemyConfig", menuName = "Config/Enemy")]
public class EnemyConfig : ScriptableObject
{
    public string enemyId;
    public int hp;
    public float speed;
    public GameObject prefab;
}
  • 优势：直接在 Editor 中配置，支持引用 Unity 资源（Prefab、贴图），校验方便。
  • 劣势：数据量大时占用内存（因为 Unity 对象的 C++ 侧开销）；多人协作 Merge 困难（.asset 是二进制 YAML，冲突难解决）。
  • 混合方案：核心数值用 Excel/CSV → ScriptableObject 只存引用型资源，运行时构建数据对象。

Mono vs IL2CPP

Mono IL2CPP
本质 JIT 编译（运行时将 IL 编译为机器码） AOT 编译（构建时 IL → C++ → 平台 native）
iOS 支持 不支持（Apple 禁止 JIT） 支持
内存 较小 较大（C++ 生成更多代码）
启动速度 快 慢（要加载完整 native 库）
运行性能 好（JIT 可热点优化） 更好（AOT 无 JIT 开销）
C# 版本支持 受限 跟随 Unity 更新

现状：Unity 2021+ 新项目 IL2CPP 为默认后端，Mono 逐渐退出主流。手游发布必须 IL2CPP。

HybridCLR（热更新方案）

  • 解决的问题：IL2CPP 下无法动态加载和执行 C# 代码，希望像 Lua/ILRuntime 一样热更新游戏逻辑。
  • 原理：在 IL2CPP 产物上补充一个解释器 + AOT 补充元数据，运行时加载 Assembly-CSharp.dll 字节码解释执行。
  • 优势：
    • 直接用 C# 写热更代码，不需要 Lua 桥接层。
    • 性能优于纯 IL 解释方案（HybridCLR 对热路径有优化）。
  • 接入成本：需要修改 il2cpp 的构建流程（有官方工具链），配置 AOT 补充元数据列表。

分层架构实践

Core (纯 C#, 不依赖 Unity)
 ├── Utils (扩展方法, 数学)
 ├── Data (数据模型, 配置解析)
 └── Systems (逻辑系统)
 
Unity Layer
 ├── MonoBehaviours (生命周期, 序列化)
 ├── Views (UI, 表现)
 └── Editor Tools

原则：逻辑层不放 Unity 特定 API（Debug.Log 除外），便于在 Editor 外做单元测试或运行仿真。
