移动端 Shader 优化速查

2019-06-07 1.1k 字
Unity GLSL

精度选择

精度 位数 范围 移动端建议
float 32-bit (全精度) 完整 IEEE 754 必要时才用
half 16-bit (中精度) -65504 ~ 65504 大部分计算首选
fixed 约 11-bit (低精度) -2.0 ~ 2.0 颜色、单位向量 
// ❌ 移动端尽量避免
float color = tex2D(...).rgb; // 全精度纹理采样

// ✅ 推荐
half3 color = tex2D(...).rgb;  // 颜色值用 half
half roughness;                 // 0-1 范围用 half
fixed3 normal = normalize(...); // 归一化后用 fixed

关键原则：值域小用 half/fixed，位置/UV/世界空间坐标才用 float。移动端 GPU（Mali、Adreno）对 half 的吞吐比 float 高很多。

Shader 变体控制

multi_compile vs shader_feature

关键字 行为 变体生成
#pragma multi_compile 所有变体都打入包体 N 个关键字 = 2^N 个变体
#pragma shader_feature 仅被材质引用的变体打入包体 按需裁剪 
#pragma shader_feature _ NORMAL_MAP
#pragma shader_feature _ EMISSION_MAP
#pragma shader_feature_local _ DETAIL_MAP

建议：能用 shader_feature 就不用 multi_compile，减少包体。运行时不可动态开关 shader_feature（只能通过材质引用），需要动态切换的场景才用 multi_compile。

变体数量爆炸

2 个 multi_compile × 3 个 shader_feature × 2 个 multi_compile_local
= 2 × 2³ × 2 = 32 个变体（仅一个 keyword 组合）

skip_variants — 剔除不需要的变体

#pragma skip_variants DIRECTIONAL SHADOWS_SCREEN

Shader 变体数查询：Project 窗口选中 Shader → Inspector → “Compiled and code” 按钮 → 看 Total variants。

discard / clip 的代价

// ❌ 移动端慎用
if (alpha < 0.5) discard;

问题：discard / clip() 导致 Early-Z 失效。GPU 无法在片段着色器执行前判断该像素是否可见，必须完整执行后才能丢弃，浪费计算。

替代方案：

  • 用 Alpha Test 代替手动 clip（部分 GPU 有硬件 Early-Z 绕过）。
  • 改为 Alpha Blend（Blend SrcAlpha OneMinusSrcAlpha），虽然 Overdraw 增加，但 GPU 可以 Early-Z 剔除。
  • 使用 Alpha to Coverage（AlphaToMask On），在 MSAA 下效果自然。

分支与动态循环

// ❌ 基于动态值的分支（GPU 两边都会执行）
if (someDynamicValue > 0.5)
    heavyCalculation();
else
    otherHeavyCalculation();

// ✅ 基于常量或 uniform 的分支（编译时优化）
if (_UseNormalMap)
    ...

// ✅ 静态分支（不产生变体，由关键字控制）
#pragma multi_compile _ _USE_NORMAL_MAP
#ifdef _USE_NORMAL_MAP
    ...
#endif

循环同理：

// ❌ 动态循环次数
for (int i = 0; i < _DynamicCount; i++)
    ...

// ✅ 固定次数或 UNITY_UNROLL
UNITY_UNROLL for (int i = 0; i < 4; i++)
    ...

纹理采样优化

  • MipMap：距离远时自动采低分辨率 Mip，减少纹理缓存 miss。几乎所有纹理都开。
  • 各向异性过滤：地面、墙面等倾斜视角的纹理开 Aniso Level 2-4，移动端不超 4。
  • 通道打包：多个灰度遮罩合成一张 RGBA 纹理（R=Metal, G=Smoothness, B=AO, A=DetailMask）。
  • Mip Map Streaming（URP/HDRP）：根据相机距离动态加载合适的 Mip Level，减少内存。
// C# 侧启用 Mip Map Streaming
QualitySettings.streamingMipmapsActive = true;

移动端带宽与填充率

瓶颈 定位方法 优化方向
带宽（Bandwidth） GPU Profiler 看 Read bytes 降低纹理分辨率、用 ASTC、减少采样次数
填充率（Fillrate） 缩小分辨率帧率明显提升 减少 Overdraw、缩小透明面积、用 Simple Shader

移动端填充率远弱于桌面。一个 1080p 屏幕有 2M 像素，每个像素执行一个 fullscreen pass 就是 2M 次片段计算。后处理叠 3 个 Pass = 6M 次。

Shader 常见写法效率对比

写法 效率 说明
normalize(v) 较低 有平方根运算
dot(a,b) 高 一次 MAD 指令
pow(x, 2) 低 会走通用指数函数
x * x 高 纯乘法，直接替换 pow(x,2)
saturate(x) 高 clamp(x, 0, 1) 的硬件加速版
step(x, y) 极高 单指令，带条件判断时比 if 高效
smoothstep 中 两次乘法 + 三次 MAD
tex2Dlod 高 明确 LOD 级别，比 tex2D 少一次导数计算
UNITY_BRANCH 低 提示编译器动态分支，移动端少用 
// ❌
half spec = pow(NdotH, _Gloss);

// ✅
half spec = NdotH;  // 先用简单近似
spec = spec * spec;
spec = spec * spec; // 连续乘法模拟 Blinn-Phong

渲染路径对比（移动端）

路径 光照限制 性能 适用
Forward 无（但实时灯多负担重） 一般 通用场景
Forward+ 大量灯光高效 较好 URP 中端+设备
Deferred 不限灯光数 带宽压力大 不建议移动端

移动端用 Forward/Forward+，不做 Deferred（G-Buffer 带宽和 MRAM 压力过大）。

URP 配置速查

Project Settings → Graphics → Scriptable Render Pipeline Settings → URP Asset
设置项 移动端建议值
Main Light Per Pixel（逐像素光照）
Additional Lights Per Vertex 或 Disabled
Cast Shadows 仅主光源
Shadow Resolution 1024
Shadow Distance 30m 以内
Cascade Count 1（不级联）
HDR Disabled（省带宽）
MSAA 2x 或关闭
Render Scale 0.8 - 1.0（性能不足时降）
