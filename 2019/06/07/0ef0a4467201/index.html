C# 与 Unity 内存管理速查

2019-06-07 799 字
Unity C#

值类型 vs 引用类型

类型 分配位置 回收方式
值类型 (struct, int, float, bool) 栈 / 作为引用类型字段时在堆 作用域退出自动释放
引用类型 (class, string, 数组, delegate) 托管堆 GC 回收

Unity 特殊规则：从 UnityEngine.Object 派生的类型（GameObject、MonoBehaviour、Texture 等）由 Unity 的 C++ 侧持有，C# 侧只存句柄。Destroy 只标记 C++ 对象，C# 包装对象待 Resources.UnloadUnusedAssets 或场景切换时释放。

GC 触发条件

  1. 堆内存分配超过阈值：Mono 和 IL2CPP 的 GC 阈值不同，分配累积到一定量触发。
  2. System.GC.Collect() 手动调用：应避免在运行时调用。
  3. 内存不足：移动端系统全局内存压力。

IL2CPP 的 GC 与 Mono 不同——使用 Boehm GC（非分代），每次触发都是全堆扫描，耗时更长。

常见 GC 分配源及优化

string 拼接

// ❌ 每帧 1 次分配
string s = "Score: " + score;

// ✅ 无分配
int score;
// UI Text 直接设数值，或
StringBuilder sb = new StringBuilder(64);
sb.Clear();
sb.Append("Score: ");
sb.Append(score);

装箱拆箱

// ❌ 装箱：int → object
object obj = 42;

// ❌ 常见隐藏装箱：Debug.Log 多参数
Debug.Log("Value: " + value); // 仍然是 string 拼接

// ❌ 枚举作为 Dictionary 的 key 不使用 comparer
Dictionary<MyEnum, int> dict; // 每次查找装箱
// ✅
Dictionary<MyEnum, int> dict = new Dictionary<MyEnum, int>(new MyEnumComparer());

foreach 旧版本陷阱

  • Unity 早期用的 Mono 编译器对 foreach 会产生垃圾（枚举器分配）。
  • IL2CPP + C# 9+ 对值类型集合 (List<T>, array) 的 foreach 已优化，无分配。
  • 问题仍存在：IEnumerable<T> 接口遍历依然装箱。

List / Dictionary 扩容

  • List<T> 初始容量 4，Add 超出容量时扩容为 2 倍，内部数组重新分配 + 拷贝 → GC。
  • Dictionary<K,V> 同理。
  • 预知数量时指定容量：
var list = new List<int>(128);
var dict = new Dictionary<string, int>(100);

Instantiate 与 Destroy

  • Instantiate：在 C++ 侧创建对象，C# 侧创建包装 → 产生分配。
  • Destroy：仅标记，不立即释放 C# 包装。
  • 高频创建/销毁场景 → 使用对象池而非直接 Instantiate/Destroy。

对象池

public class SimplePool<T> where T : Component
{
    private Queue<T> pool = new Queue<T>();
    private T prefab;
    private Transform parent;

    public SimplePool(T prefab, Transform parent, int prewarm = 10)
    {
        this.prefab = prefab;
        this.parent = parent;
        for (int i = 0; i < prewarm; i++)
        {
            var obj = Object.Instantiate(prefab, parent);
            obj.gameObject.SetActive(false);
            pool.Enqueue(obj);
        }
    }

    public T Get()
    {
        T obj = pool.Count > 0 ? pool.Dequeue() : Object.Instantiate(prefab, parent);
        obj.gameObject.SetActive(true);
        return obj;
    }

    public void Return(T obj)
    {
        obj.gameObject.SetActive(false);
        obj.transform.SetParent(parent);
        pool.Enqueue(obj);
    }
}

要点：预热池避免运行时分配峰值；SetActive 控制可见性；SetParent 保持层级干净。

Resources.UnloadUnusedAssets

  • 作用：释放所有未被引用的 UnityEngine.Object 的 C# 包装。
  • 调用时机：场景加载完成后。
  • 代价：遍历所有对象标记 + 清理，主线程阻塞，通常耗时几百毫秒。
  • 不要在游戏过程中反复调用。

内存分析切入点

  • Unity Profiler → Memory 模块：看 Total Allocated、GC Allocated per Frame。
  • Memory Profiler 包：Snapshot 对比，揪出未释放的 Texture、Mesh 引用。
  • **GC.GetTotalMemory(false)**：纯 C# 托管内存总量，可在代码中打点。
  • 常驻泄漏源：静态事件未取消订阅、OnDestroy 未移除回调、DontDestroyOnLoad 累积。
